1. Systemöversikt — vad pratar med vad?

👤
Konsultfirma (användare)
Webbläsare
Laddar upp RFP:er, hanterar konsultprofiler, får anbudsutkast. All interaktion sker via webbgränssnittet.
M0: Ladda upp RFP → se strukturerad analys
M1: + Hantera konsultprofiler, se matchresultat
M2: + Generera och redigera anbudsutkast
↕ HTTPS (krypterat)

Next.js M0
Frontend + Backend i ett · Vercel (prod) · localhost (dev)
Hela vår applikation. Next.js är både frontend och backend — renderar webbsidor OCH kör server-logik (API routes). Hostad på Vercel i produktion.
Frontend = det användaren ser (formulär, knappar, resultatvyer)
Backend = API routes som tar emot data, pratar med Claude och Supabase
💡 Tänk: en restaurang där kocken och servitören är samma person

📄 src/app/api/ — backend-logik

↕ API-anrop
(skickar text, får JSON)

(läs/skriv data)

Claude API MOLN
Anthropic · api.anthropic.com
AI-hjärnan. Vår app skickar text till Claude, får tillbaka strukturerad analys. Claude lagrar ingenting — den processar och glömmer.
🔑 Autentisering via API-nyckel i .env.local
📤 Vi skickar: systemprompt + RFP-text (eller CV:n, eller instruktioner)
📥 Vi får: strukturerad JSON tillbaka
💡 Claudes API är stateless — den kommer inte ihåg tidigare anrop
💰 Vi betalar per token (in + ut). Se kostnadstabell nedan.
🔒 Data skickas krypterat, lagras inte av Anthropic (API-policy)
🗄️
Supabase MOLN
PostgreSQL + Storage · supabase.co
Vår databas och fillagring i molnet. Tänk: Google Drive (för filer) + Excel (för strukturerad data) — fast programmerbart och säkert.
Två delar:
📦 Storage = fillagring. Här hamnar originalfiler (Word, PDF, etc.)
🗃️ Database = PostgreSQL. Här lagras strukturerad data (analyser, profiler, anbud)
🔑 Autentisering via service role key (server-side, aldrig synlig för användaren)
💡 Supabase är en hanterad tjänst — vi sköter ingen server
🔒 Data krypterad at rest, anslutning via SSL
📍 Hostad i EU (Frankfurt) — bra för GDPR
↑ M3: proaktiv scanning av externa källor
📡
RFP-radar M3
Schemalagd scanning · Haiku + Sonnet
Skannar upphandlingsplattformar automatiskt, matchar mot firmans konsultpool, rekommenderar de bästa möjligheterna. Flippar modellen från reaktiv till proaktiv.
Flöde: Ny RFP upptäcks → Haiku snabbanalys → matcha mot konsultpool → Go/No-Go → notifiera firman
Nyckelinsikt: "Det finns 12 nya upphandlingar denna vecka. 3 matchar era konsulter. Här är den med 78% vinstchans."
💡 Kombinerat med Go/No-Go: "Skippa den du tittade på — den här som kommer på tisdag passar er bättre"

3. Vad lämnar kunden över? Och vad byggs upp över tid?

📤 Kunden laddar upp / matar in

  • KRÄVS Förfrågningsunderlag (RFP) — Word, PDF eller text. Det offentliga/privata underlaget de vill svara på.
  • KRÄVS Konsult-CV:n — Fullständiga CV:n för sina konsulter. Lagras internt, aldrig exponerat rakt av.
  • VALFRITT Tidigare anbud — Vinnande anbud de skickat förut. Hjälper AI:n lära sig firmans ton och stil.
  • VALFRITT Firmaprofil & mallar — Standardtexter, boilerplate, certifieringar, referensuppdrag.
  • VALFRITT Prismodell / timpriser — Om de vill att anbudet ska innehålla prisuppskattning.

🔄 Systemet genererar

  • GENERERAD Strukturerad RFP-analys — Krav, vikter, deadlines, red flags. Maskinläsbar JSON.
  • GENERERAD Konsultmatchning — Rankad lista med poäng och motivering per konsult vs krav. M1
  • GENERERAD Anbudsutkast — Komplett anbudstext, redo att redigeras. M2
  • GENERERAD Kuraterade konsultprofiler — Externt lager: sammanfattade profiler anpassade per anbud. M1

📈 Byggs upp över tid (plattformsvärde)

  • ACKUMULERAS Kravmönster — Vilka krav som är vanliga i vilka branscher/upphandlingar. Inget annat verktyg har denna data.
  • ACKUMULERAS Matchningshistorik — Vilka kompetenskombinationer vinner vilka typer av uppdrag.
  • ACKUMULERAS Anbudsutfall — Om kunden markerar "vann" / "förlorade" kan vi lära oss vad som funkar.
  • ACKUMULERAS Branschbenchmarks — Genomsnittligt antal krav, vanliga vikter, typiska red flags per sektor.
  • ACKUMULERAS Marknadsöversikt — Vilka typer av uppdrag publiceras var, frekvens, säsongsmönster, genomsnittlig konkurrens. M3

🔒 Datahantering & sekretess

  • Tvålagers CV-modell: Internt lager (full CV, bara AI:n ser) → Externt lager (kuraterad profil, syns i anbud)
  • Claude API: Stateless — data processas men lagras inte av Anthropic
  • Supabase: Krypterad lagring, EU-hosting (Frankfurt), GDPR-kompatibel
  • Isolering: Varje firma ser bara sin egen data. Aggregerad anonymiserad data används för plattformsinsikter.

4. Hur blir systemet bättre över tid?

Vi tränar inte Claude — det går inte med API:t. Istället bygger vi bättre kontext för varje anrop. Databasen växer, men prompten hålls liten. Det är skillnaden.

📚
Historikbaserad kontext
Promptinjection från databas
Vid varje anrop hämtar vi relevanta tidigare resultat och skickar med. Opus "lär sig" genom att se firmans bästa exempel.
Anbudsskrivning: "Här är 3 tidigare anbud som vann i liknande upphandlingar"
Matchning: "Konsult X fick 92 poäng på liknande uppdrag förra kvartalet"
RFP-analys: "I senaste 20 upphandlingarna inom vård var vanligaste red flag otydlig tidsplan"
💡 Ingen magi — SQL-query → data i prompten → bättre svar
🏆
Utfallsloop
Vann / Förlorade → feedback
Kunden markerar om anbudet vann eller inte. Över tid ser vi vilka formuleringar, strukturer och matchningar som korrelerar med vinst.
Kräver bara ett fält: bids.outcome: 'won' | 'lost' | 'pending'
Vinnande anbud blir "goda exempel" i framtida prompter
Förlorande anbud = anti-patterns att undvika
💡 Varje anbud gör nästa lite bättre — flyghjulseffekt
🎨
Firmspecifik stilguide
Extraheras en gång, sparas
Istället för att skicka 50 gamla anbud för att "lära tonen", extraherar vi en stilguide en gång. ~500 tokens som aldrig växer.
Ton: "formell men inte byråkratisk"
Undvik: "leverera", "synergi", "best practice"
Föredra: "genomföra", "samverkan", "beprövad metod"
Struktur: sammanfattning först, metodik, team, pris
💡 Byggs upp organiskt — kan redigeras av kunden

Kontextbloat-problemet — och lösningen

En firma med 200 anbud kan inte skicka allt till Claude — det blir 2M tokens och $300 per anrop. Lösningen: trattformad kontextfiltrering. Databasen är stor, prompten är liten.

Lager 1: Databas (SQL)
200 anbud → WHERE bransch = 'vård' AND utfall = 'vann' → 15 anbud
Kostnad: $0 — ren SQL-filtrering
▼
Lager 2: Sammanfattning
Aggregerad statistik: "7 av 10 vinnande anbud betonade förändringsledning"
Kostnad: $0 — förberäknade aggregat
▼
Lager 3: AI-filtrering (Haiku)
15 anbud → "vilka 3 är mest relevanta?" → 3 anbud
Kostnad: ~$0.01
▼
Prompt till Opus
~4 000 tokens: kravanalys + 3 exempel + stilguide + statistik
Kostnad: ~$0.35 (Opus)

Samma promptstorlek oavsett om firman gjort 10 eller 1 000 anbud. Databasen växer, prompten gör det inte.

5. Dataflöde: Från RFP till anbud

1
Kund laddar upp RFP
Word/PDF/text → webbformulär → sparas i Supabase Storage (original) + documents-tabellen (metadata)
2
Dokument parsas till text
mammoth.js extraherar text ur Word. Ren filhantering i Next.js — ingen AI, inga tokens, ingen extern tjänst.
Kostnad: $0
3
RFP analyseras → strukturerad JSON M0
Text skickas till Claude API → får tillbaka krav, vikter, kompetenser, red flags. Resultatet sparas i analyses-tabellen i Supabase.
Sonnet: ~10 000 tokens in → ~2 000 tokens ut ≈ $0.06
4
Konsulter matchas mot krav M1
Strukturerad JSON (inte rå-RFP) + CV:n från consultants-tabellen skickas till Claude. Haiku filtrerar, Sonnet matchar. Resultat sparas i matches-tabellen.
Haiku: ~$0.04 (filter) + Sonnet: ~$0.11 (matchning) ≈ $0.15
5
Go/No-Go-beslut M1.5
Sonnet analyserar matchresultat + historiska utfall → estimerad vinstchans, styrkor, luckor, alternativa möjligheter. Sparas i go_no_go-tabellen. Firman beslutar: satsa eller skippa?
Sonnet: ~3 000 tokens in → ~1 500 tokens ut ≈ $0.03
6
Anbud genereras (om Go) M2
Opus får komprimerad kontext (~3 500 tokens): kravanalys + matchade konsulter + firmans mallar + go/no-go-insikter. Skriver anbudsutkast → bids-tabellen.
Opus: ~3 500 tokens in → ~4 000 tokens ut ≈ $0.35
7
Senior konsult granskar & skickar
Människan i loopen — redigerar utkastet, godkänner, skickar in. Ingen AI i detta steg.
8
Utfall rapporteras (vann/förlorade)
Kunden markerar utfall i bids.outcome. Kopplas tillbaka till go_no_go-prediktionen → feedback-loop som förbättrar framtida vinstprediktioner.
Total pipeline: ~$0.59 per anbud (inkl. Go/No-Go)
