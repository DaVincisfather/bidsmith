# M1.5: Go/No-Go Agent — Design Spec

**Date:** 2026-04-06
**Status:** Approved (verbal)
**Depends on:** M0 (RFP analysis), M1 (consultant matching)
**Prepares for:** M2 (bid generation), M3 (RFP radar)

---

## Purpose

A consulting firm that receives an RFP must decide: **invest 4–8 hours writing a bid, or pass?** The Go/No-Go agent provides a structured, data-driven recommendation in ~30 seconds — surfacing strengths, gaps, win probability, and actionable improvement suggestions.

The agent also establishes the **feedback pipeline** that improves predictions over time: every bid outcome (won/lost) feeds back into future Go/No-Go assessments.

## User Flow

```
RFP uploaded → Analysis (M0) → Matching (M1) → User tweaks team via instant swap
→ User clicks "Lock team" → Go/No-Go runs automatically → Results shown
→ User can: "Back to team" (swap, re-lock, re-run) OR "Proceed to bid" (always available)
```

**Key behaviors:**
- "Lock team" triggers the analysis — no separate button needed
- User can run Go/No-Go multiple times with different team compositions
- Each run is saved (1:N relationship) for comparison and training data
- "Proceed to bid" is never gated — user always decides

## Go/No-Go View

Displayed as a new section below the team proposal on the analysis page.

### 1. Must-requirement check
Binary pass/fail per ska-krav. If ANY fails → win_probability = 0%. Prominently displayed.

### 2. Win probability
Percentage (0–100) with short reasoning. At 0%, the blocking must-requirement is highlighted.

**Without historical data:** Sonnet estimates based on requirement coverage, team strength, evaluation criteria weights, and red flags. UI shows "AI estimate" label to be transparent.

**With historical data (future):** Prompt includes outcomes from similar past RFPs (same domain/sector). Accuracy improves over time.

### 3. Strengths
Bullet list of what the team does well, tied to specific RFP requirements.

### 4. Gaps
What's missing, tied to specific requirements.

### 5. Improvement suggestions (actionable)
Concrete swap recommendations referencing the full consultant pool:
- "Replace [consultant A] with [consultant B] → estimated +X%"
- "You're missing [competency] for must-requirement [Y]. Do you have a consultant with this experience?"
- Sorted by estimated impact (highest first)

### 6. Recommendation
`go` | `no-go` | `go-with-reservations` + reasoning text.

### 7. Actions
- "Back to team" — unlocks team, user can swap and re-lock
- "Proceed to bid" — always enabled regardless of recommendation

### 8. History (if multiple runs exist)
Simple comparison: "Run 1: Team A — 72% | Run 2: Team B — 58%". Lets user see impact of team changes.

## Architecture

### Approach: Single Sonnet call

One API call receives the full context:
- RFP analysis (requirements, evaluation criteria, red flags)
- Locked team with individual scores and reasoning from M1
- Rest of consultant pool with scores (for improvement suggestions)
- Domain tag for future historical lookups

~3000 tokens in, ~1500 tokens out. Cost: ~$0.03 per run.

### API Endpoint

`POST /api/go-no-go`

**Input:**
```typescript
{
  analysisId: string;
  teamConsultantIds?: string[];  // Optional — defaults to top-matched team if omitted (M3 compatibility)
}
```

When `teamConsultantIds` is omitted, the endpoint selects the highest-scored consultant per required level from the match result. This enables M3's automated pipeline to trigger Go/No-Go without human intervention.

**Output:** `GoNoGoResult` (see Data Model below)

### Prompt Design

System prompt instructs Sonnet to:
1. Check each must-requirement against team competencies/references — binary pass/fail
2. If any must-requirement fails, set winProbability to 0
3. Assess should-requirements and nice-to-haves for the probability estimate
4. Weight evaluation criteria as specified in the RFP analysis
5. Factor in red flags
6. Generate improvement suggestions by comparing team gaps against available consultants in the pool
7. Estimate impact of each suggested swap

## Data Model

### GoNoGoResult (JSON stored in DB)

```typescript
interface GoNoGoResult {
  mustRequirements: {
    requirement: string;
    met: boolean;
    coveredBy: string | null;  // Consultant name, null if unmet
  }[];
  winProbability: number;        // 0–100, forced to 0 if any must-req unmet
  winProbabilityReasoning: string;
  strengths: string[];
  gaps: string[];
  improvements: {
    swap: { remove: string; add: string };  // Consultant names
    swapIds: { removeId: string; addId: string };  // Consultant UUIDs
    estimatedImpact: string;     // e.g. "+15%"
    reason: string;
  }[];
  recommendation: "go" | "no-go" | "go-with-reservations";
  reasoning: string;
}
```

### DB Table: `go_no_go_assessments`

| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK | |
| analysis_id | FK → analyses | |
| organization_id | FK → organizations | |
| team_consultant_ids | uuid[] | The locked team |
| result | jsonb | GoNoGoResult |
| decision | enum: pending/go/no-go | User's actual decision |
| decision_at | timestamptz | When user decided |
| created_at | timestamptz | |

**Relationship:** analyses 1:N go_no_go_assessments (multiple runs with different teams).

### M0 Extension: Domain tag in RfpAnalysis

Add `domain` field to the RFP analyzer output:

```typescript
// Added to RfpAnalysis interface
domain: string;  // e.g. "IT", "management", "finance", "public-sector-it"
```

This is extracted by Sonnet during RFP analysis (M0) at near-zero marginal cost — one extra field in the existing prompt. Enables efficient historical lookup for the feedback loop: "How did this firm perform on similar domain RFPs?"

## Feedback Pipeline

### Phase 1 (M1.5 — now)
- `decision` column captures whether user proceeded to bid or backed out
- Each Go/No-Go run is saved with team composition and result
- Domain tag enables future grouping

### Phase 2 (M2 — future)
- `bids.outcome` (won/lost/pending) links back to go_no_go_assessment
- When generating new Go/No-Go assessments, prompt can include: "For similar [domain] RFPs, this firm has won X of Y bids"
- More outcome data → better calibrated win_probability

### Data flywheel
Firms that actively report outcomes get better predictions. This creates:
- Lock-in (switching cost = losing prediction history)
- Network effect within each firm (more data = more value)
- Incentive to use the platform for all bids, not just some

## M3 Compatibility

The `teamConsultantIds` parameter is optional. When M3's RFP radar discovers a new opportunity:

1. Auto-run RFP analysis (M0)
2. Auto-run matching (M1)
3. Auto-run Go/No-Go with `teamConsultantIds` omitted → uses best-matched team
4. Notify firm: "New RFP found — 78% estimated win probability with your best team"

No changes needed to Go/No-Go at that point — just a new caller.

## Cost

~$0.03 per Go/No-Go run (Sonnet, ~3000 tokens in, ~1500 out).
Multiple runs per RFP are expected (team iteration) — budget ~$0.10 per RFP for Go/No-Go.

## Out of Scope

- Bid generation (M2)
- Historical outcome comparison in prompt (requires M2 outcome data)
- RFP radar integration (M3)
- Multi-org anonymized benchmarking
